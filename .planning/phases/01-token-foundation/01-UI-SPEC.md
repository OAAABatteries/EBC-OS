## Design System Architecture

Phase 1 uses a two-file architecture:

| File | Scope | Extension Point |
|------|-------|----------------|
| `src/tokens.css` | NEW — universal tokens (spacing, typography, touch, focus, transitions) | Imported once in `src/main.jsx` |
| `src/data/constants.js` THEMES | MODIFIED — per-theme tokens (shadows, semantic color aliases) | Added to each of the 8 theme `vars` objects |
| `src/styles.js` | MODIFIED — utility classes consuming tokens (.shadow-sm/md/lg, .transition-*, .touch-target) | Appended to existing utility section |

**Rationale:** Universal tokens (spacing, touch, transitions) must NOT be duplicated across 8 themes. Per-theme tokens (shadows, semantic colors) MUST vary because Matrix theme uses `none` shadows and Blueprint uses colored glow shadows — a universal value would break them.

**Source:** RESEARCH.md §Architecture Patterns Pattern 1 and Pattern 2

## Spacing Scale

All values in `px`. The codebase is 100% px-based — rem would create compounding issues in Capacitor WebViews where `font-size` on `html` may differ from browser default.

**Source:** REQUIREMENTS.md TOKN-01; RESEARCH.md §Pattern 1 (px vs rem rationale); CONTEXT.md D-03, D-04

| Token | CSS Variable | Value | Primary Usage in Codebase |
|-------|-------------|-------|--------------------------|
| space-1 | `--space-1` | 4px | Icon gaps, tight inline padding |
| space-2 | `--space-2` | 8px | Compact element spacing, tag padding |
| space-3 | `--space-3` | 12px | Button padding (existing .btn uses 12px) |
| space-4 | `--space-4` | 16px | Default card/section padding (most common value in styles.js) |
| space-5 | `--space-5` | 20px | Wider card padding, modal padding |
| space-6 | `--space-6` | 24px | Section vertical spacing |
| space-8 | `--space-8` | 32px | Layout gaps, larger section breaks |
| space-10 | `--space-10` | 40px | Major vertical spacing |
| space-12 | `--space-12` | 48px | Page-level spacing |

Exceptions:
- Touch target minimum: `--touch-min: 44px` — this is NOT a spacing token but an interaction token. It does not map to a space-N step intentionally (WCAG 2.5.5 requirement).
- PWA safe area: `env(safe-area-inset-*)` — these remain in `index.css` as platform values, not tokenized.

### Size and Role Map

| Role | CSS Variable | Size | Codebase Origin |
|------|-------------|------|----------------|
| xs | `--text-xs` | 10px | table headers, form labels, gantt header |
| sm | `--text-sm` | 11px | badge text, kpi-label, logo-sub, gantt bar labels |
| base | `--text-base` | 13px | primary body: table cells, buttons, toast, form-input |
| md | `--text-md` | 14px | secondary body: gantt-label, room-header, btn-icon |

| 2xl | `--text-2xl` | 20px | section-title |

**Note on 12.5px nav values:** `nav-item` and `nav-more-btn` use `12.5px` in styles.js. These round up to `--text-base` (13px) — the 0.5px difference is imperceptible and eliminates a non-standard value.

When adding typography tokens to `tokens.css`, simultaneously update `styles.js` text utility classes to consume them:

```css

.text-sm  { font-size: var(--text-sm);  }
.text-md  { font-size: var(--text-md);  }
```

This is Phase 1 scope because it is vocabulary wiring, not portal refactoring.

## Interaction Tokens

**Source:** REQUIREMENTS.md TOKN-03, TOKN-04, TOKN-05; RESEARCH.md §Pattern 1

### Touch Target

| Token | CSS Variable | Value | Applies To |
|-------|-------------|-------|-----------|
| touch-min | `--touch-min` | 44px | all interactive elements (WCAG 2.5.5) |

Current `.btn` in styles.js has `min-height: 36px` — this is below WCAG minimum. Phase 2 will apply `--touch-min` to `FieldButton`. Phase 1 defines the token only.

Utility class (added to styles.js):
```css
.touch-target { min-height: var(--touch-min); min-width: var(--touch-min); }
```

### Focus Ring

| Token | CSS Variable | Value | Notes |
|-------|-------------|-------|-------|
| focus-ring | `--focus-ring` | `0 0 0 2px var(--amber)` | Theme-adaptive: `--amber` resolves lazily at paint time — safe because App.jsx applies theme vars to `document.documentElement` before any render |

Current styles.js suppresses focus with `outline: none` on `.form-input:focus`. Phase 1 defines the token so Phase 2 components can apply it correctly. The existing suppression is NOT removed in Phase 1 — that refactor belongs to the component phases.

Utility class (added to styles.js):
```css
.focus-visible:focus-visible {
  outline: none;
  box-shadow: var(--focus-ring);
}
```

### Transitions

| Token | CSS Variable | Value | Usage |
|-------|-------------|-------|-------|
| duration-micro | `--duration-micro` | 150ms | hover states, color changes |
| duration-state | `--duration-state` | 300ms | tab switches, panel opens, modals |
| ease-standard | `--ease-standard` | `cubic-bezier(.4, 0, .2, 1)` | all transitions |
| transition-micro | `--transition-micro` | `var(--duration-micro) var(--ease-standard)` | shorthand for micro-interactions |
| transition-state | `--transition-state` | `var(--duration-state) var(--ease-standard)` | shorthand for state transitions |

Current codebase uses `0.15s`, `0.2s`, `0.25s`, `0.4s` scattered inline — all normalize to these two steps.

Utility classes (added to styles.js):
```css
.transition-micro { transition: all var(--transition-micro); }
.transition-state  { transition: all var(--transition-state); }
```

## Color

**Phase 1 produces semantic color tokens, not a new color palette.** All token values reference existing theme primitives (`--green`, `--amber`, `--red`, `--blue`, `--yellow`, `--text2`, `--text3`) which are already defined in all 8 THEMES entries.

**Source:** REQUIREMENTS.md TOKN-07; RESEARCH.md §Pattern 2; CONTEXT.md D-05, D-06

### 60/30/10 Surface Split (Reference — not changed in Phase 1)

| Role | CSS Variable | Coverage | Notes |
|------|-------------|----------|-------|
| Dominant (60%) | `--bg`, `--bg2` | Backgrounds, page surfaces | Already defined per-theme |
| Secondary (30%) | `--card`, `--bg3` | Cards, panels, nav | Already defined per-theme |
| Accent (10%) | `--amber` | Focus rings, active states, highlights | Already defined per-theme |
| Destructive | `--red` | Delete actions, denied status only | Already defined per-theme |

These primitives are existing — Phase 1 does NOT redefine them. Phase 1 adds semantic aliases on top.

### Status Semantic Aliases (added to THEMES object, per-theme)

| Token | CSS Variable | Value | Semantic Meaning |
|-------|-------------|-------|-----------------|
| status-approved | `--status-approved` | `var(--green)` | Approved, confirmed, synced |
| status-pending | `--status-pending` | `var(--amber)` | Pending, awaiting action |
| status-denied | `--status-denied` | `var(--red)` | Denied, rejected, error |
| status-in-transit | `--status-in-transit` | `var(--blue)` | In transit, in progress |
| status-project | `--status-project` | `var(--text2)` | Project-level neutral label |
| status-office | `--status-office` | `var(--text3)` | Office-level muted label |

**Theme-adaptation:** Each theme defines its own `--green`, `--amber`, `--red`, `--blue` primitives. By aliasing, `--status-approved` will automatically render as Matrix's green glow or Blueprint's cyan — no per-theme overrides needed for status colors.

### Phase Semantic Aliases (added to THEMES object, per-theme)

Named tokens (not numbered) because phases are named business concepts in EBC's domain. `--phase-active` is self-documenting; `--phase-1` is meaningless. MoreTabs.jsx chart bars are array-indexed — they do NOT need semantic tokens (leave PHASE_COLORS array untouched).

| Token | CSS Variable | Value | EBC Domain Meaning |
|-------|-------------|-------|-------------------|
| phase-active | `--phase-active` | `var(--green)` | Actively under construction |
| phase-estimating | `--phase-estimating` | `var(--amber)` | In estimating / bidding |
| phase-pre-construction | `--phase-pre-construction` | `var(--blue)` | Pre-construction planning |
| phase-completed | `--phase-completed` | `var(--text3)` | Completed, archived |
| phase-warranty | `--phase-warranty` | `var(--yellow)` | In warranty period |
| phase-in-progress | `--phase-in-progress` | `var(--green)` | Alias used in GanttScheduleView |

**Source:** CONTEXT.md D-06; RESEARCH.md §Pattern 2 (named vs numbered decision)

### Shadow Scale (per-theme, added to THEMES object)

Shadows MUST be per-theme because Matrix theme has `"--shadow": "none"` by design (flat/scanline aesthetic) and Blueprint uses colored glow shadows, not black-opacity.

| Token | CSS Variable | Standard value (steel/dark themes) |
|-------|-------------|-----------------------------------|
| shadow-sm | `--shadow-sm` | `0 1px 4px rgba(0,0,0,0.12)` |
| shadow-md | `--shadow-md` | `0 2px 12px rgba(0,0,0,0.28)` |
| shadow-lg | `--shadow-lg` | `0 4px 32px rgba(0,0,0,0.45)` |

Theme-specific overrides:
- **Matrix:** `--shadow-sm: none`, `--shadow-md: 0 0 10px rgba(0,255,65,0.04)`, `--shadow-lg: 0 0 20px rgba(0,255,65,0.08)`
- **Blueprint:** `--shadow-sm: 0 0 0 1px rgba(0,191,239,0.06)`, `--shadow-md: 0 0 12px rgba(0,191,239,0.08)`, `--shadow-lg: 0 0 24px rgba(0,191,239,0.12)`
- **Daylight (light theme):** Values use lighter opacity: `0 1px 4px rgba(0,0,0,0.06)`, `0 2px 8px rgba(0,0,0,0.12)`, `0 4px 20px rgba(0,0,0,0.18)`

Utility classes (added to styles.js):
```css
.shadow-sm { box-shadow: var(--shadow-sm); }
.shadow-md { box-shadow: var(--shadow-md); }
.shadow-lg { box-shadow: var(--shadow-lg); }
```

**Source:** CONTEXT.md D-07, D-08; RESEARCH.md §Common Pitfalls Pitfall 1

## Validation Method

No automated test runner exists in this codebase. Phase 1 validation is manual browser inspection.

| Requirement | Validation Step |
|-------------|----------------|
| TOKN-01 Spacing | `getComputedStyle(document.documentElement).getPropertyValue('--space-4')` in DevTools — should return `" 16px"` |
| TOKN-02 Typography | DevTools computed styles: verify `--text-base` returns `13px`, `--text-3xl` returns `28px` |
| TOKN-03 Touch | Inspect `.touch-target` element — computed min-height should show `44px` |
| TOKN-04 Focus | Tab through interactive elements in each theme — amber ring should appear on focus |
| TOKN-05 Transitions | Hover a button — micro-interaction should feel snappy (150ms); open a modal — state change at 300ms |
| TOKN-06 Shadows | Side-by-side: `.shadow-sm`, `.shadow-md`, `.shadow-lg` must be visibly distinct in each theme |
| TOKN-07 Semantic colors | Switch all 8 themes — `--phase-active` must render the theme's green; `--status-denied` must render the theme's red |

**Phase gate before `/gsd:verify-work`:** Full 8-theme visual pass — all tokens resolve non-empty, no broken vars.
